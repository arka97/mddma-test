# Storage, Media & Uploads

Everything the platform stores in Lovable Cloud Storage: buckets, paths, size limits, validation, and how the frontend wires uploads.

## Buckets

| Bucket | Public read | Write | Owner |
|---|---|---|---|
| `avatars` | yes | owner | The user themselves |
| `company-assets` | yes | company owner | Logos, covers, brand gallery |
| `product-images` | yes | company owner | Product cover, gallery (≤3), product video |
| `ad-assets` | yes | **admin only** | Sponsored ad creative |

All four are public-read so URLs can be embedded in the SPA without signed links. Public read is **only** safe because nothing private (KYC docs, invoices, personal IDs) is ever stored here.

## Path conventions

Uploads use `${userId}/${timestamp}-${random}.${ext}` so:

- Each user's files live in their own folder (helps debugging, avoids name collisions).
- The timestamp prevents cache stomp when re-uploading.
- The random suffix avoids collisions within the same millisecond.

The path is generated by `src/lib/storage.ts → uploadFile()`. Don't construct paths by hand.

## Size limits & MIME allowlist

Source of truth: `src/lib/storage.ts`.

| Kind | Max size | Allowed MIME |
|---|---|---|
| Image | **10 MB** | `image/jpeg`, `image/png`, `image/webp`, `image/gif` |
| Video (only on `product-images`) | **100 MB** | `video/mp4`, `video/webm`, `video/quicktime` |

**SVG is explicitly rejected** — both by MIME and by `.svg` filename suffix — because SVG can embed JavaScript.

```ts
// src/lib/storage.ts (excerpt)
const MAX_IMAGE_BYTES = 10 * 1024 * 1024; // 10 MB
const MAX_VIDEO_BYTES = 100 * 1024 * 1024; // 100 MB

export class UploadValidationError extends Error {}
```

`validateFile()` throws an `UploadValidationError` with a human-readable message like `"File is too large (max 100 MB)."` or `"SVG files are not allowed."`. UI code should catch and toast.

## Frontend usage

```ts
import { uploadFile } from "@/lib/storage";

const url = await uploadFile("product-images", user.id, file);
if (!url) {
  // validation already logged; toast a generic error
  toast({ title: "Upload failed", variant: "destructive" });
  return;
}
```

`uploadFile` returns `null` on validation OR storage error to keep callers backward-compatible. To surface the **exact** validation reason (recommended for new UI), call `validateFile` directly and catch:

```ts
try {
  validateFile(file, /* allowVideo */ true);
} catch (e) {
  if (e instanceof UploadValidationError) toast({ title: e.message });
}
```

`ProductsPage.tsx` already does this for the bulk + edit video flows.

## Gallery limit (DB-enforced)

The `enforce_product_gallery_limit` trigger refuses any `INSERT/UPDATE` on `products` where `gallery` length > 3. With the cover image, that caps a product at 4 visual assets. UI should validate before submit; the trigger is the safety net.

## Public URL strategy

```ts
const { data } = supabase.storage.from(bucket).getPublicUrl(path);
return data.publicUrl;
```

Public URLs are stable as long as the path doesn't change. To "delete" an asset, remove the path from the parent row first (so nothing references it), then call `supabase.storage.from(bucket).remove([path])` to free space.

## Recommended folder layout per bucket

| Bucket | Convention |
|---|---|
| `avatars` | `<userId>/<timestamp>-<rand>.<ext>` |
| `company-assets` | `<userId>/<timestamp>-<rand>.<ext>` (one folder per owner; one company per owner today) |
| `product-images` | `<userId>/<timestamp>-<rand>.<ext>` |
| `ad-assets` | `<userId>/<timestamp>-<rand>.<ext>` (admin user IDs only) |

There is no per-product subfolder — the `products` row references the URL, so reorganisation isn't required.

## Operational recipes

### Replace a product image
1. Upload new file via `/account/products` → variant edit. New URL is written to `products.image_url`.
2. Old URL is now orphaned. Periodic cleanup: query `storage.objects` against `products.image_url` / `products.gallery` to identify orphans.

### Bulk-clear orphans (admin SQL)
```sql
-- Files in product-images bucket NOT referenced by any product row
WITH refs AS (
  SELECT image_url AS url FROM products WHERE image_url IS NOT NULL
  UNION
  SELECT unnest(gallery) FROM products WHERE gallery IS NOT NULL
  UNION
  SELECT video_url FROM products WHERE video_url IS NOT NULL
)
SELECT name FROM storage.objects
WHERE bucket_id = 'product-images'
  AND NOT EXISTS (SELECT 1 FROM refs WHERE refs.url LIKE '%' || storage.objects.name);
```

### Replace an ad image
Admins go to `/account/moderation` → Ads. Upload writes to `ad-assets`, then sets `advertisements.image_url`. Old image becomes orphan; clean periodically.

## Failure modes & how they surface

| Failure | Where it surfaces |
|---|---|
| File > 10 MB image / 100 MB video | `UploadValidationError` toast |
| SVG submitted | `UploadValidationError` toast: "SVG files are not allowed." |
| Wrong MIME (e.g. `.heic`) | `UploadValidationError` toast: "Unsupported file type" |
| RLS denial (e.g. non-admin uploading to `ad-assets`) | Console error, `uploadFile` returns `null` |
| Network failure | Console error, `uploadFile` returns `null` |
| 4th gallery image | DB trigger `22023` exception — UI should catch |

## Out of scope

- KYC document storage. Currently the project does not host KYC PDFs / images at all — verification is admin-driven via `/account/moderation`. If KYC ever moves in-app, create a private `kyc-docs` bucket with owner-only read AND admin read, and never list in this doc as `public read`.
- Signed URLs / expiring links. Not used because every bucket is public-read.
- Image resizing. Lovable Cloud Storage does not transform on the fly; oversize images render at full resolution. Resize client-side before upload if perf matters.
